# 租赁

------

# 一、租赁域到底做什么

租赁域就只做两件事：

## 1）发布资源

资源拥有者把农场资源发布出来，供别人租。

## 2）订阅资源

普通用户查看资源并订阅，后续可以续订或退订。

------

# 二、租赁功能范围

## 做

- 发布资源
- 查看资源详情
- 编辑资源

### 订阅端

  这些在原始租赁说明里都已经明确砍掉了。

# 七、租赁域业务规则

这是最重要的部分，代码必须按这个写。

------

## 资源规则

1. 只有发布者本人能修改和下架资源
2. 只有 `ON_SHELF` 资源能被订阅
3. 下架后不能新增订阅
4. 下架不影响已有有效订阅
5. 如果当前存在有效订阅，建议只允许修改描述类字段，不允许修改价格 / 类型 / 面积等核心字段。

------

## 订阅规则

1. 用户不能订阅自己发布的资源
2. 同一资源同一时间只能有一个 `ACTIVE` 订阅
3. 只有 `ACTIVE` 订阅能续订
4. 只有 `ACTIVE` 订阅能退订
5. 退订后不可续订
6. 到期后状态自动变 `EXPIRED`。

# 八、租赁域数据库设计

------

## 1）资源表 `resource`

```sql
CREATE TABLE resource (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    owner_user_id BIGINT NOT NULL,
    title VARCHAR(100) NOT NULL,
    resource_type VARCHAR(30) NOT NULL,
    area DECIMAL(10,2) NOT NULL,
    location_desc VARCHAR(255) NOT NULL,
    price_per_month DECIMAL(10,2) NOT NULL,
    min_lease_months INT NOT NULL,
    description VARCHAR(1000),
    status VARCHAR(20) NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);
```

### 索引

```sql
CREATE INDEX idx_resource_owner ON resource(owner_user_id);
CREATE INDEX idx_resource_status_type ON resource(status, resource_type);
```

------

## 2）订阅表 `subscription`

```sql
CREATE TABLE subscription (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    resource_id BIGINT NOT NULL,
    tenant_user_id BIGINT NOT NULL,
    start_date DATE NOT NULL,
    end_date DATE NOT NULL,
    status VARCHAR(20) NOT NULL,
    cancelled_at DATETIME NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);
```

### 索引

```sql
CREATE INDEX idx_subscription_resource_status ON subscription(resource_id, status);
CREATE INDEX idx_subscription_tenant_status ON subscription(tenant_user_id, status);
CREATE INDEX idx_subscription_resource_time ON subscription(resource_id, start_date, end_date);
```

这些表结构和索引方案都来自你租赁文档里的最简版。

### 请求

处理逻辑：

3. 默认状态设为 `ON_SHELF`

5. 调用 `ResourceRepository.insert`

### Repository：insert

执行：

```sql
INSERT INTO resource (...)

```

------

### 数据库结果

`resource` 新增一条记录：

```text

owner_user_id=1
title=A区大棚1号
```

------

### 返回结果


## 流程 4：订阅资源

这是租赁域最核心的流程。

### 前端

用户在资源详情页点击“订阅”，弹出弹窗，填写：

- 起始日期
- 租期（月）

系统自动算结束日期。

这个交互在原方案里已经明确建议用弹窗，不单独做页面。

------

### 请求

```http
POST /api/subscriptions
Content-Type: application/json
{
  "resourceId": 1001,
  "tenantUserId": 2001,
  "startDate": "2026-04-15",
  "leaseMonths": 3
}
```

------

### Controller：SubscriptionController.create

负责：

- 接收 DTO
- 做基础参数校验
- 调用 `SubscriptionService.createSubscription`

------

### Service：createSubscription

处理逻辑：

1. 查资源是否存在
2. 校验资源状态必须是 `ON_SHELF`
3. 校验不能订阅自己的资源
4. 校验租期是否满足 `minLeaseMonths`
5. 校验该资源当前没有别的 `ACTIVE` 订阅
6. 计算 `endDate`
7. 创建 `Subscription`
8. 状态设为 `ACTIVE`
9. 写入数据库

这条规则就是租赁文档里的核心规则：
**同一资源同一时间只允许一个有效订阅。**

------

### Repository

#### ResourceRepository.findById

查资源

#### SubscriptionRepository.findActiveByResourceId

查这个资源是否已有有效订阅

#### SubscriptionRepository.insert

插入订阅记录

------

### 数据库结果

`subscription` 新增记录：

```text
id=9001
resource_id=1001
tenant_user_id=2001
start_date=2026-04-15
end_date=2026-07-14
status=ACTIVE
```

------

### 返回结果

```json
{
  "id": 9001,
  "resourceId": 1001,
  "tenantUserId": 2001,
  "startDate": "2026-04-15",
  "endDate": "2026-07-14",
  "status": "ACTIVE"
}
```

------


# 十、状态机

## 资源状态机

```text
ON_SHELF -> OFF_SHELF
```

如果你后面想支持重新上架，再加：

```text
OFF_SHELF -> ON_SHELF
```

但 MVP 先不做，原文档也说可以只保留单向。

------

## 订阅状态机

```text
ACTIVE -> CANCELLED
ACTIVE -> EXPIRED
```

不要允许：

```text
CANCELLED -> ACTIVE
EXPIRED -> ACTIVE
```

重新租就新建一条订阅，不回滚旧记录。这个和原方案一致。
