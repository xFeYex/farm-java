# 七、农产品交易模块业务规则

这是代码必须写死的部分。

------

## 商品规则

1. 只有卖方本人能修改和下架商品
2. 只有 `ON_SHELF` 商品能被购买
3. 下架后不能新增订单
4. 下架不影响已有订单
5. 如果商品已经有未完成订单，建议只允许修改描述、封面图，不允许改价格
6. 如果没有进行中订单，可以修改价格、库存、描述

------

## 订单规则

1. 买方不能买自己的商品
2. 下单必须校验库存
3. 扣库存和创建订单必须在同一事务
4. 发货前状态必须是 `CREATED`
5. 收货前状态必须是 `SHIPPED`
6. 只有已收货订单才能申请退货
7. 退货申请必须在收货后 7 天内发起
8. 超过 7 天且未退货，订单自动完成

------

## 结算规则

1. 卖家不能在买家未收货前收到钱
2. 卖家不能在 7 天无理由退货期内收到钱
3. 买家发起退货后，该订单结算状态关闭
4. 只有超过 7 天且无退货的订单才能结算

------

## 流程 8：7 天后自动完成订单

这是后台定时任务，不是前端触发，但业务上必须有。

------

### 触发方式

例如每小时或每天跑一次任务。

------

### Service：autoFinishOrders

处理逻辑：

查询所有满足条件的订单：

- `status = RECEIVED`
- `settle_status = PENDING`
- `can_settle_at <= now()`

然后：

1. 更新订单状态为 `FINISHED`
2. 写入 `finishedAt`
3. 更新：
   - `settle_status = SETTLED`
   - `settled_at = now()`

原方案里这一步是通过定时任务把结算单改为 `SETTLED`，把订单改为 `FINISHED`。最终版只是把两者都写在订单表里。

------

### Repository：findPendingCanSettleList + updateById

------

# 十、状态机

------

## 商品状态机

```text
ON_SHELF -> OFF_SHELF
```

原方案提到也可支持重新上架，但 MVP 可以只做单向。

------

## 订单状态机

```text
CREATED -> SHIPPED
SHIPPED -> RECEIVED
RECEIVED -> RETURNING
RETURNING -> RETURNED
RECEIVED -> FINISHED
CREATED -> CANCELLED
```

不要允许：

```text
RETURNED -> FINISHED
FINISHED -> RETURNING
```

这和原方案一致。

------

## 结算状态机

最终版虽然不单独建表，但状态机保留：

```text
PENDING -> SETTLED
PENDING -> CLOSED
```

来源仍然是原方案的结算状态设计。

## 功能

- 买方：查看商品、购买、确认收货、申请退货、查看我的订单
- 卖方：上架、查看 / 编辑 / 下架商品、查看卖方订单、发货、完成退货
- 系统：7 天自动完成订单并完成结算

## MVC

- `OrderController`
- `ProductService`
- `OrderService`

## 关键规则

- 只有上架商品能被购买
- 下单扣库存和创建订单必须同事务
- 收货后开始算 7 天
- 7 天内可申请退货
- 超过 7 天且无退货，自动完成订单并结算
- 退货后结算关闭
