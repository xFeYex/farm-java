# 智慧生产模块（新增版）

## 一、最终功能范围

在原有能力基础上，扩成这 4 个功能：

1. **种植计划制定**
   - 像写日志一样记录自己的种植计划
   - 可新建、查看、编辑、删除
   - 本质是“计划记录”
2. **物联网设备联动**
   - 继续保留假数据模式
   - 可查看设备参数
   - 可修改设备参数并保存
   - 用于页面展示和联调，不做真实设备控制
3. **种植状态监测**
   - 继续保留原来的环境数据查看能力
   - 展示温度、湿度、光照、土壤湿度等
   - 可刷新最新数据
4. **收获管理**
   - 用户确认本次收获的农产品
   - 收获信息存数据库
   - 后续农产品共享模块直接查库使用
   - 这里先不强做跨模块接口

这和你现在说的目标是一致的，而且也符合系统总文档里“生产 → 农产品产出 → 手动上架”的弱联动思路。

------

# 二、保持不变的部分

这些不动：

## 1）不改模块定位

智慧生产域仍然是：

- 只能在租赁有效后使用
- 依赖 `subscription.status = ACTIVE`
- 不自己处理租赁逻辑

这个在总说明和智慧生产原文档里已经定死。

## 2）不改原 MVC 主骨架

继续保留：

- `ProductionController`
- `ProductionOrchestrationController`
- `ProductionViewService`
- `ProductionOrchestrationService`

原有环境数据、摄像头、编排配置三块继续能用。

## 3）不改原访问控制逻辑

继续通过租赁模块的 `SubscriptionRepository.findActiveByResourceId` 做访问校验，不新增独立权限系统。

# 六、接口设计

原来有 4 个接口：

- `GET /api/production/dashboard/{resourceId}`
- `PUT /api/production/orchestration/{resourceId}`
- `GET /api/production/environment/{resourceId}`
- `GET /api/production/camera/{resourceId}`

现在建议扩成下面这 8 个，原接口保留，新增接口补上。

------

## 1）查询智慧生产主页数据

```http
GET /api/production/dashboard/{resourceId}
```

返回：

- 模块可用状态
- 当前种植计划摘要
- 设备参数摘要
- 环境数据
- 摄像头信息
- 收获记录摘要

------

## 2）修改物联网设备/编排参数

沿用原接口，扩一下含义即可：

```http
PUT /api/production/orchestration/{resourceId}
```

可修改：

- 每日次数
- 每次时长
- 启动时间
- 设备参数（如喷水器频率、持续时间、阈值）

这样你不用新增一个全新设备 Controller，直接在原编排控制里兼容“假设备参数”。这最符合“基础架构不改动”。

# 十、数据库 SQL 设计

## 1）种植计划表

```sql
CREATE TABLE planting_plan (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    resource_id BIGINT NOT NULL,
    subscription_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    title VARCHAR(100) NOT NULL,
    plan_content TEXT NOT NULL,
    plan_date DATE NOT NULL,
    status VARCHAR(20) NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);

CREATE INDEX idx_plan_resource_date ON planting_plan(resource_id, plan_date);
CREATE INDEX idx_plan_user_status ON planting_plan(user_id, status);
```

## 2）收获记录表

```sql
CREATE TABLE production_harvest (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    resource_id BIGINT NOT NULL,
    subscription_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    product_name VARCHAR(100) NOT NULL,
    category VARCHAR(50) NOT NULL,
    harvest_quantity DECIMAL(10,2) NOT NULL,
    unit VARCHAR(20) NOT NULL,
    harvest_date DATE NOT NULL,
    remark VARCHAR(500),
    status VARCHAR(20) NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);

CREATE INDEX idx_harvest_resource_date ON production_harvest(resource_id, harvest_date);
CREATE INDEX idx_harvest_user_status ON production_harvest(user_id, status);
```

# 十一、完整业务流程

## 流程 1：制定种植计划

### 前端

用户进入智慧生产主页，在“种植计划”区域点击新增。

### 请求

```http
POST /api/production/plan/101
Content-Type: application/json
{
  "userId": 2001,
  "title": "番茄种植计划",
  "planContent": "4月18日播种，4月20日检测湿度，4月22日补水。",
  "planDate": "2026-04-18"
}
```

### Controller

```
ProductionPlanHarvestController.createPlan
```

### Service

```
PlantingPlanService.createPlan
```

处理逻辑：

1. 校验用户对该资源有有效订阅
2. 组装计划对象
3. 状态默认 `ACTIVE`
4. 写入数据库

### Repository

```
PlantingPlanRepository.insert
```

------

## 流程 2：物联网设备联动

### 前端

用户修改设备参数，例如喷水器每天几次、每次几分钟。

### 请求

```http
PUT /api/production/orchestration/101
Content-Type: application/json
{
  "userId": 2001,
  "timesPerDay": 3,
  "durationMinutes": 10,
  "startTimes": ["08:00", "14:00", "20:00"],
  "sprinklerEnabled": true,
  "targetHumidity": 70
}
```

### Service

仍然走 `ProductionOrchestrationService.updateSimpleConfig`

处理逻辑：

1. 校验有效订阅
2. 查当前编排配置
3. 更新白名单字段
4. 设备参数可以写进 `config_json`

这样和原先的“简单参数修改”完全兼容。原文档本来就允许修改简单编排参数，只是不做复杂规则引擎。

------

## 流程 3：状态监测

### 前端

用户进入主页或点击刷新。

### 请求

```http
GET /api/production/environment/101?userId=2001
```

### Service

仍然走 `ProductionViewService.getEnvironmentView`

处理逻辑：

1. 校验有效订阅
2. 查最新一条环境快照
3. 返回温度、湿度、光照、土壤湿度

这和原智慧生产设计完全一致。

# 十二、关键业务规则

## 访问规则

1. 只有 `subscription.status = ACTIVE` 才能进入智慧生产模块。
2. 只有当前订阅人本人可以查看和修改。
3. 订阅失效后，计划、设备、状态、收获都不可操作。

## 种植计划规则

1. 计划本质是日志式记录，不做复杂审批流。
2. 只允许当前订阅人新增和编辑。
3. 计划删除建议做逻辑删除或改状态，不建议真删。

## 设备联动规则

1. 设备数据允许是假数据。
2. 只允许修改白名单参数。
3. 不做真实硬件控制，不做消息下发，不做复杂联动引擎。

## 状态监测规则

1. 只查最新一条环境快照。
2. 摄像头离线只返回状态，不报业务异常。这个原方案已经定了。

## 收获管理规则

1. 只有当前订阅有效时才能登记收获。
2. 收获记录一旦生成，建议允许查看、允许备注修改，但不建议随意删除。
3. 收获数据作为共享模块基础数据源，不在这里做交易逻辑。
